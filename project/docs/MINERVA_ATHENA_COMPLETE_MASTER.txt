2. Ingestion: Universal Multilingual OCR & Translation Stack

To support documents in 136+ languages (Hindi, Japanese, Arabic, German, Tamil,
Cyrillic, etc.) and low-quality photos/handwritten documents, Minerva uses a
three-tier pipeline:

1. Optical Character Recognition (OCR):
   * Digital PDF Parser: For text-based PDFs, extracting clean UTF-8 text,
     keeping table formats intact.
   * Vision OCR Engine: For photos, scans, and handwriting, using Google Cloud
     Vision API and PaddleOCR (which supports multi-language and complex
     vertical/right-to-left layout analysis).
2. Semantic Translation & Sanitization:
   * An intermediate LLM layer identifies the input language and translates it
     into a uniform schema (English/Target Language) while preserving academic
     symbols, formulas (LaTeX), and scientific terms.
3. Hierarchy Builder:
   * Converts the parsed content into structured JSON. It handles page numbers,
     index tables, and footnotes without mixing them into the study content.

4. End-to-End Operational Flow (No-Breakdown Execution)

Step 1: The Builder Discussion (Ingestion & Sync)
* Action: The student uploads a 500-page textbook PDF or a photo of their
  handwritten notes in Japanese and asks: "Help me prepare for my university
  exam."
* Process: The system uploads the file via chunked HTTP stream to prevent
  browser timeout. The backend processes the PDF page-by-page.
* Result: The system starts a Socratic discussion: "I have scanned your
  document. I see 8 main themes. Let's start with Theme 1. Would you like a
  quick overview or should we dive straight in?"

Step 2: Roadmap Generation & Auto-Synchronization
* Action: The system maps out the study roadmap.
* Logic: Nodes are color-coded by priority (High = Red, Medium = Orange,
  Low = Green). If the subject is Mathematics, the roadmap auto-syncs equations
  to show pre-requisites (e.g. Derivatives node must be completed to unlock
  Integrals).
* Progress Save: The state is saved page-by-page in MongoDB so that if the
  user closes the app, they resume at the exact same node.

Step 3: Deep Socratic Study & Verification Deck
* Action: The student reviews the node.
* Dynamic Content: For a PhD student, it shows peer-reviewed citations. For a
  3rd grader, it shows animated steps.
* YouTube Ingestor: Fetches videos from verified APIs and validates that they
  are active (no dead links).
* Verifications: The AI poses a question: "Explain why we need a stabilizer
  for this reaction?" The student submits their answer. The AI grades it. If
  they struggle, the path self-heals by adding a supporting sub-node with a
  simplified guide.

Step 4: The Spaced Repetition Analytics Dashboard
* Minerva monitors their learning retention based on the Ebbinghaus Forgetting
  Curve.
* It logs their performance in a Cognitive Strength & Weakness Matrix:
  * Strong Topics: High recall speed, low error rate (retention time increased).
  * Weak Topics: Repeated errors (retention flag set; scheduled for review).

Step 5: Weakness-Adaptive Exam Deck & Downloadable PDF
* Action: User clicks "Generate Practice Exam."
* Generation Engine: The system checks the database and builds an exam paper.
  If they got Topic A wrong twice, the exam will have 3 questions on Topic A.
* PDF Builder: Using a PDF generation library (like pdfkit or reportlab on the
  backend), it generates a clean, printable exam paper with space for writing
  answers, matching the exact format of CBSE/University boards.

6. Self-Healing & System Resiliency (Zero-Failure Policy)

To ensure the system never crashes or breaks under heavy load:
1. Queueing System: Large PDF processing tasks are sent to a background worker
   queue (using Redis/Celery) so that the HTTP connection never times out.
2. File Size Optimization: Large PDFs are split into 10-page chapters and
   processed asynchronously.
3. Graceful Fallbacks: If the Google Vision OCR fails due to rate limits, the
   system automatically falls back to Tesseract OCR or PaddleOCR locally on
   the worker.
4. Offline Sync: Student progress is cached in browser localStorage and synced
   with MongoDB whenever the network connection is stable.

2. Core Groundbreaking Pillars (Unique Ideas)

Pillar A: Hierarchical Semantic Chunking (Tree of Knowledge)
Standard AI systems split PDFs into random blocks of 1000 characters, which
breaks paragraphs and destroys context. Project Athena uses a Hierarchical

   (Chapters > Major Topics > Sub-topics > Core Definitions).
2. Dependency Mapping: It determines which topics are precursors. For example,
   if the PDF is about Web Development, the system maps that HTML DOM must be
   learned and verified before React Hooks can unlock.
3. Priority Indexing: The AI automatically scans for words like "Important",
   "Frequently Asked", or weightage marks in the syllabus to assign High,
   Medium, or Low Priority to each node.

------------------------------------------------------------------------

Pillar B: The "Infinite Zoom" Study Deck
Instead of giving the user a static, boring text summary, we introduce
Multi-Fidelity Zoom-In Study Cards:
* Level 1: Summary Zoom (10% depth): A bulleted executive summary of the topic.
* Level 2: Standard Zoom (50% depth): Detailed theory, clear analogies, and
  key formulas.
* Level 3: Microscopic Zoom (100% depth): Step-by-step mathematical proofs,
  code-level execution blocks, edge cases, and common misconceptions.
* Socratic Dialogue Assistant: An inline chat interface where a student can
  highlight any text, click "Drill Me", and trigger a personalized Q&A loop.
  The AI asks questions rather than giving answers, forcing the student to
  think deeply.

------------------------------------------------------------------------

Pillar C: Live-Verified YouTube Link Ingestion
Most AI engines hallucinate dead video links. Project Athena uses a Real-Time
Video Validation Agent:
1. When a node is generated (e.g., "Binary Search Tree Rotation"), the backend
   runs a search query against YouTube's Search API.
2. It fetches the top 3 highest-rated educational videos (from verified
   channels like freeCodeCamp, Khan Academy, 3Blue1Brown).
3. Verification Hook: The server makes a quick headless check to ensure the
   video status is HTTP 200 (not dead/private) and parses the video transcript
   to fetch the exact timestamps matching the sub-topic.

------------------------------------------------------------------------

Pillar D: Micro-Tasks & Study Journal PDF Export
For every node, the student must complete a set of Actionable Tasks:
* Active Recall Prompts: "Explain this concept to a 10-year-old," or "Write
  a pseudo-code implementation."
* Submission Engine: The student types or uploads their code/written work.
* AI Evaluation Rubric: The AI evaluates the task, gives a score, saves it to
  their Profile Database, and unlocks the next node.
* Study Journal PDF: At any point, the student can download their study log as
  a beautifully styled PDF containing all lessons learned, their task
  submissions, and AI feedback.

------------------------------------------------------------------------

Pillar E: Weakness-Weighted Diagnostic Exams
When the user finishes a block of topics, they generate a test inside the Exam
Generator:
* Weakness Weighting: The exam generator does not pick questions randomly. It
  checks their performance logs and allocates 60% of the questions to topics
  where the student made mistakes during micro-tasks.
* Format Matching: It designs the paper according to their selected
  Board/Standard (e.g., CBSE, ICSE, University grading schemas) with Marks
  Distribution (e.g., Section A: 1-mark questions, Section B: 5-mark long
  questions).

------------------------------------------------------------------------

Pillar F: The "Grand Finale" Subject Exam
Once the entire roadmap is traversed and completed:
1. The system locks the learning deck and unlocks the Grand Finale Exam.
2. This is a comprehensive, timed examination covering the most critical and
   high-yield questions from the entire PDF.
3. Upon submission, it provides a Neural Assessment Report showing which areas
   they scored highest in, time spent per section, and generates a printable
   Certificate of Completion.

------------------------------------------------------------------------

B1. MASTER SYSTEM WORKFLOW — THE COMPLETE USER JOURNEY
=======================================================

PHASE 0: STUDENT ARRIVES (ONBOARDING)
--------------------------------------
  Student visits Minerva platform (web or mobile app)
  |
  +-- [NEW USER] --> Diagnostic Interview (30 questions, adaptive)
  |                  --> System measures:
  |                       * Prior knowledge (what do you already know?)
  |                       * Learning style (visual / verbal / kinesthetic)
  |                       * Cognitive speed (response latency)
  |                       * Motivation type (fear-based / goal-based)
  |                       * Language preference (136 options)
  |                  --> OUTPUT: Student Cognitive Fingerprint (SCF)
  |
  +-- [RETURNING USER] --> Load saved SCF from database
                          --> Resume from last node position
                          --> Run Spaced Repetition Injector
                              (show due review cards before new content)

PHASE 1: DOCUMENT INGESTION
-----------------------------
  Student uploads PDF / Image / Scan / Handwritten Note
  |
  +--> Chunked HTTP upload (no timeout, progress bar shown)
  +--> Backend Queue (Redis/Celery) receives job
  +--> OCR Pipeline activates:
       |
       +-- Digital PDF?  --> PyMuPDF / pdfminer extracts UTF-8 text
       +-- Image/Scan?   --> Google Cloud Vision API (primary)
                              PaddleOCR (fallback, offline)
                              Tesseract (fallback 2)
       +-- Handwriting?  --> Vision API handwriting model
  |
  +--> Language Detection (LangDetect + LLM verification)
  +--> Translation to working language (DeepL / Google Translate API)
  +--> Hierarchy Builder:
       Chapter > Section > Topic > Subtopic > Definition > Formula > Example
  |
  +--> OUTPUT: Structured JSON document ready for Curriculum Engine

PHASE 2: ROADMAP GENERATION
-----------------------------
  Curriculum Engine receives structured JSON
  |
  +--> Dependency Graph Builder:
       * Which topic requires which prerequisite?
       * Example: Calculus --> Algebra (Calculus is locked until Algebra done)
  |
  +--> Priority Indexer:
       * Scan for "important", "frequently asked", "must know", marks weight
       * Assign: HIGH / MEDIUM / LOW priority to every node
  |
  +--> Cognitive Matrix applies student grade level:
       * Grade 1-5   : Gamified, emoji, stories
       * Grade 6-12  : Structured, board-exam format
       * UG/PG       : Professional, case-study format
       * PhD         : Academic, citation-based format
  |
  +--> Roadmap UI renders:
       * Tree-of-knowledge visual (nodes and edges)
       * Color coding: Red = High Priority, Orange = Medium, Green = Low
       * Lock/Unlock state per node (based on prerequisites)
  |
  +--> OUTPUT: Interactive roadmap displayed to student

PHASE 3: LEARNING LOOP (CORE ENGINE)
--------------------------------------
  Student selects an unlocked node
  |
  +--> ZOOM LEVEL SELECTION:
       * Summary (10%): Bullet summary, 2-3 lines per sub-topic
       * Standard (50%): Full theory, analogies, formulas
       * Deep Dive (100%): Proofs, edge cases, common misconceptions
  |
  +--> SOCRATIC DIALOGUE:
       * Student can highlight any sentence and click "Drill Me"
       * AI asks a probing question rather than giving the answer
       * Student must respond before proceeding
  |
  +--> YOUTUBE VERIFICATION:
       * Backend searches YouTube API for the exact topic
       * Verifies link is active (HTTP 200 check)
       * Parses transcript to find exact timestamp for sub-topic
       * Displays top 3 verified video cards to student
  |
  +--> MICRO-TASK DECK:
       * 2-5 tasks generated per node
       * Types: explain in own words / write code / solve problem /
                draw diagram / fill formula / compare concepts
       * Student submits task (text / code / image upload)
  |
  +--> AI ASSESSMENT:
       * Task graded by LLM rubric (0-100 score)
       * Feedback given inline (what was right, what was wrong, how to fix)
       * Score logged to Cognitive Dashboard
  |
  +--> NODE COMPLETION CHECK:
       * All micro-tasks passed (score >= 70%)? --> Node marked COMPLETE
       * Score < 70%? --> Remedial sub-node injected automatically
                          Student must clear remedial before retrying main
  |
  +--> SPACED REPETITION INJECTION:
       * Completed node added to SR database
       * Review scheduled: Day+1, Day+3, Day+7, Day+15, Day+30
       * Each review: a quick 3-question recall test
       * Passed: interval doubles
       * Failed: interval resets, new healing path injected

PHASE 4: ASSESSMENT & EXAM GENERATION
---------------------------------------
  Student completes a block of topics (or requests exam)
  |
  +--> WEAKNESS ANALYSIS:
       * System queries all performance logs for this subject
       * Identifies which topics have error_rate > 30%
       * Identifies which topics have recall_speed < threshold
  |
  +--> EXAM PAPER CONSTRUCTION:
       * 60% questions from weak topics
       * 30% questions from medium performance topics
       * 10% questions from strong topics (as confidence builders)
       * Format matches selected board (CBSE/ICSE/University/Custom)
       * Sections: MCQ / Short Answer / Long Answer / Case Study / Essay
  |
  +--> GRAND FINALE EXAM (after 100% roadmap completion):
       * Comprehensive paper covering all HIGH priority nodes
       * Timed exam (configurable by teacher or student)
       * Upon submission: Full Neural Assessment Report generated
       * Certificate of Completion generated with digital signature
  |
  +--> PDF EXPORT:
       * Study Journal PDF (all notes, tasks, AI feedback)
       * Exam Paper PDF (printable, board-format)
       * Report Card PDF (Cognitive strengths/weaknesses, parent-ready)

PHASE 5: PARENT & TEACHER PORTAL
----------------------------------
  Parallel to student journey:
  |
  +--> PARENT PORTAL:
       * Real-time progress: % roadmap completed, last active time
       * Cognitive Report Card (monthly auto-generated)
       * Alert: "Your child has been struggling with Trigonometry for 3 days"
       * Communication channel with Minerva AI (parent can ask questions)
  |
  +--> TEACHER COMMAND BRIDGE:
       * Class roster view: all students, their progress, risk scores
       * Predictive alert: "Student X has 92% probability of failing mid-term"
       * Push custom roadmap to any student with one click
       * Download class-wide performance report as CSV/PDF
       * Create cohort-specific exam papers

PHASE 6: WEEKLY CHECKPOINT (TUITION CLASS MODE)
-------------------------------------------------
  Every Saturday/Sunday (or teacher-configured day):
  |
  +--> System locks new content
  +--> Initiates Weekly Checkpoint Session
       * Oral/chat-based viva conducted by AI teacher
       * Questions drawn from the week's completed nodes
       * Student responds verbally (voice-to-text) or via chat
       * AI grades response quality (concept accuracy + depth)
  |
  +--> Weekly performance summary generated
  +--> Next week's customized syllabus auto-scheduled
  +--> Parent notified with weekly progress digest

G1. HIERARCHICAL SEMANTIC CHUNKING ALGORITHM
=============================================

PURPOSE: Convert raw OCR text into structured Knowledge Node tree.

ALGORITHM: HierarchicalSemanticChunker

INPUT:
  raw_text   : String   (full document text, UTF-8)
  subject    : String
  grade_level: Enum

OUTPUT:
  node_tree  : Array<KnowledgeNode>  (hierarchical tree)

PSEUDOCODE:

  FUNCTION chunk_document(raw_text, subject, grade_level):

    // STEP 1: Split into segments by structural markers
    segments = split_by_patterns(raw_text, patterns=[
      "^Chapter \d+",           // Chapter headers
      "^\d+\.\d+ ",             // Numbered sections (1.1, 2.3)
      "^[A-Z][A-Z ]+$",         // ALL CAPS headings
      "^(SECTION|UNIT|PART) ",  // Keyword headers
      "^(Note|Important|Remember|Definition):",  // Special markers
    ])

    // STEP 2: Classify each segment
    classified = []
    FOR EACH seg IN segments:
      level = classify_level(seg)  // CHAPTER | SECTION | TOPIC | SUBTOPIC | LEAF
      classified.append({text: seg, level: level})

    // STEP 3: Build tree
    tree = []
    stack = []  // Tracks current path in hierarchy

    FOR EACH item IN classified:
      WHILE stack is not empty AND stack.top.level >= item.level:
        stack.pop()

      node = create_node(
        title    = extract_title(item.text),
        content  = extract_body(item.text),
        parent   = stack.top if stack else null,
        level    = item.level
      )

      IF stack is not empty:
        stack.top.children.append(node)
      ELSE:
        tree.append(node)  // Root chapter node

      stack.push(node)

    // STEP 4: Priority assignment
    FOR EACH node IN flatten(tree):
      node.priority = compute_priority(node.content)
      // Uses E3 priority weighting table

    // STEP 5: Dependency mapping
    FOR EACH node IN flatten(tree):
      node.dependencies = detect_prerequisites(node.content, tree)
      // Checks if content references other nodes

    RETURN tree

  FUNCTION compute_priority(text):
    text_lower = lowercase(text)
    IF any(word IN text_lower FOR word IN ["most important", "frequently asked"]):
      RETURN HIGH   // weight >= 0.90
    IF any(word IN text_lower FOR word IN ["very important", "remember", "note"]):
      RETURN MEDIUM // weight 0.50 - 0.89
    RETURN LOW      // weight < 0.50

G2. COGNITIVE DIGITAL TWIN (CDT) ALGORITHM
============================================

PURPOSE: Build and update the student's cognitive profile during first
         5 interactions, and continuously refine it.

ALGORITHM: CognitiveTwinBuilder

INPUT:
  interaction_log : Array<InteractionEvent>
  student_id      : UUID

OUTPUT:
  scf             : StudentCognitiveFingerprint

PSEUDOCODE:

  FUNCTION build_cognitive_twin(student_id):

    interactions = get_first_N_interactions(student_id, N=5)

    // MEASURE 1: Response Latency
    latencies = [event.response_time_ms FOR event IN interactions]
    avg_latency = mean(latencies)

    IF avg_latency < 3000:    processing_speed = FAST
    ELIF avg_latency < 7000:  processing_speed = AVERAGE
    ELIF avg_latency < 15000: processing_speed = SLOW
    ELSE:                     processing_speed = VERY_SLOW

    // MEASURE 2: Vocabulary Tier (Type-Token Ratio + rare word frequency)
    all_words = concatenate all student text responses
    type_token_ratio = unique_words(all_words) / total_words(all_words)
    rare_word_rate   = count(words with frequency < 0.001 in corpus) / total_words

    IF rare_word_rate > 0.15 AND type_token_ratio > 0.65: vocab = EXPERT
    ELIF rare_word_rate > 0.05 AND type_token_ratio > 0.50: vocab = ADVANCED
    ELIF type_token_ratio > 0.40: vocab = INTERMEDIATE
    ELSE: vocab = BASIC

    // MEASURE 3: Semantic Coherence (LLM-based)
    coherence_scores = []
    FOR EACH response IN student_open_ended_responses:
      prompt = f"Score the logical coherence of this student answer from 0 to 1.
                 Answer: {response}
                 Return JSON: {{score: float}}"
      result = call_llm(prompt)
      coherence_scores.append(result.score)

    avg_coherence = mean(coherence_scores)

    // COMPUTE ZPD OFFSET
    // Zone of Proximal Development = content 10% harder than student's current level
    // Adjust based on performance
    zpd_base = 0.10  // 10% harder by default

    IF avg_coherence > 0.80 AND processing_speed == FAST:
      zpd_offset = 0.20  // Student can handle more challenge
    ELIF avg_coherence < 0.40 OR processing_speed == VERY_SLOW:
      zpd_offset = 0.05  // Reduce challenge, keep student in flow
    ELSE:
      zpd_offset = zpd_base

    // DETECT BURNOUT THRESHOLD
    error_spike = detect_error_rate_spike(student_id, window_minutes=120)
    latency_spike = detect_latency_spike(student_id, window_minutes=120)

    IF error_spike > 2.0 AND latency_spike > 2.5:  // 200% and 250% increase
      trigger_break_prompt(student_id)

    // BUILD SCF
    scf = StudentCognitiveFingerprint(
      student_id       = student_id,
      processing_speed = processing_speed,
      vocabulary_tier  = vocab,
      semantic_coherence = avg_coherence,
      zpd_offset       = zpd_offset,
      response_latency_ms = avg_latency,
      version          = 1
    )

    save_to_db(scf)
    RETURN scf

PURPOSE: Select content difficulty 10% above student's current verified level
         to keep them in FLOW STATE without boredom or frustration.

  FUNCTION select_next_node_difficulty(student_id, subject):

    // Get student's current mastery level for this subject
    mastery_history = get_node_scores(student_id, subject, last=10)
    current_mastery = mean(mastery_history)  // 0.0 to 1.0

    // Get ZPD offset from SCF
    scf = get_scf(student_id)
    zpd_offset = scf.zpd_offset  // typically 0.10

    // Target difficulty
    target_difficulty = current_mastery + zpd_offset

    // Clamp to valid range
    target_difficulty = max(0.0, min(1.0, target_difficulty))

    // Find node closest to target difficulty
    candidate_nodes = get_unlocked_nodes(student_id, subject)
    best_node = argmin(abs(node.difficulty - target_difficulty)
                       FOR node IN candidate_nodes)

    RETURN best_node

  // UPDATE after each task completion:
  FUNCTION update_difficulty_after_task(student_id, task_result):

    IF task_result.score >= 90:
      // Student aced it, increase challenge
      scf.zpd_offset = min(scf.zpd_offset + 0.02, 0.25)

    ELIF task_result.score >= 70:
      // Passed normally, maintain offset
      // No change

    ELIF task_result.score < 50:
      // Student struggling, reduce challenge
      scf.zpd_offset = max(scf.zpd_offset - 0.03, 0.03)

    save_scf(scf)

G4. EBBINGHAUS SPACED REPETITION ALGORITHM (SM-2 ENHANCED)
============================================================

PURPOSE: Schedule optimal review times for every completed node to
         combat the Ebbinghaus Forgetting Curve and maximize long-term
         memory retention.

ALGORITHM: SpacedRepetitionAutoInjector (SM-2 enhanced)

PSEUDOCODE:

  FUNCTION schedule_review(node_id, student_id, quality_score):
    // quality_score: 0-5 rating of recall quality

    record = get_sr_record(node_id, student_id)

    IF quality_score >= 3:  // Correct recall
      IF record.repetitions == 0:
        new_interval = 1
      ELIF record.repetitions == 1:
        new_interval = 3
      ELSE:
        new_interval = round(record.interval_days * record.ease_factor)

      // Update ease factor (SM-2 formula)
      new_ef = record.ease_factor + (0.1 - (5 - quality_score) *
                (0.08 + (5 - quality_score) * 0.02))
      new_ef = max(1.3, new_ef)  // Minimum ease factor = 1.3

      record.repetitions += 1
      record.ease_factor = new_ef
      record.interval_days = new_interval
      record.next_review_date = TODAY + new_interval days

    ELSE:  // Failed recall (score 0, 1, or 2)
      record.repetitions = 0
      record.interval_days = 1
      record.next_review_date = TOMORROW
      // Inject immediate remedial node into roadmap
      inject_remedial_node(node_id, student_id)

    save_sr_record(record)

  FUNCTION get_todays_review_queue(student_id):
    due_records = query_db(
      student_id = student_id,
      next_review_date <= TODAY,
      order_by = priority DESC,
      limit = 20  // Max 20 reviews per session
    )
    RETURN due_records

G5. WEAKNESS-WEIGHTED EXAM GENERATION ALGORITHM
=================================================

PURPOSE: Generate a customized exam paper where harder/weakly-known
         topics receive proportionally more questions.

ALGORITHM: WeaknessWeightedExamGenerator

PSEUDOCODE:

  FUNCTION generate_exam(session_id, exam_config):

    // Step 1: Get all topic performance data
    performance_log = get_all_task_scores(session_id)

    topic_stats = {}
    FOR EACH entry IN performance_log:
      topic = entry.topic
      IF topic NOT IN topic_stats:
        topic_stats[topic] = { scores: [], attempts: 0 }
      topic_stats[topic].scores.append(entry.score)
      topic_stats[topic].attempts += 1

    // Step 2: Classify topics
    weak_topics   = []  // error_rate > 30% OR avg_score < 60
    medium_topics = []  // error_rate 15-30% OR avg_score 60-80
    strong_topics = []  // error_rate < 15% AND avg_score > 80

    FOR EACH topic, stats IN topic_stats:
      avg_score = mean(stats.scores)
      error_rate = count(s < 70 FOR s IN stats.scores) / len(stats.scores)

      IF avg_score < 60 OR error_rate > 0.30:
        weak_topics.append({topic: topic, weight: 3.0})
      ELIF avg_score < 80 OR error_rate > 0.15:
        medium_topics.append({topic: topic, weight: 1.5})
      ELSE:
        strong_topics.append({topic: topic, weight: 0.5})

    // Step 3: Calculate question slots per tier
    total_questions = exam_config.total_questions  // e.g. 30 questions

    weak_slots   = round(total_questions * 0.60)  // 60% = 18 questions
    medium_slots = round(total_questions * 0.30)  // 30% = 9  questions
    strong_slots = total_questions - weak_slots - medium_slots  // 10% = 3

    // Step 4: Distribute questions within each tier (by weight)
    weak_distribution   = weighted_distribute(weak_topics,   weak_slots)
    medium_distribution = weighted_distribute(medium_topics, medium_slots)
    strong_distribution = weighted_distribute(strong_topics, strong_slots)

    // Step 5: Generate questions using LLM
    questions = []
    ALL_DISTRIBUTIONS = weak_distribution + medium_distribution + strong_distribution

    FOR EACH (topic, count, difficulty) IN ALL_DISTRIBUTIONS:
      FOR i IN range(count):
        q = generate_question_llm(
          topic      = topic,
          difficulty = difficulty,
          format     = select_format(exam_config.sections, i),
          board      = exam_config.board
        )
        questions.append(q)

    // Step 6: Organize into sections
    sections = build_sections(questions, exam_config.sections_config)

    // Step 7: Render PDF
    pdf_url = render_pdf(sections, exam_config.metadata)

    RETURN ExamPaper(sections=sections, pdf_url=pdf_url, ...)

G6. YOUTUBE LINK VALIDATION ALGORITHM
=======================================

PURPOSE: Find real, verified YouTube videos for each learning node —
         zero hallucinated or dead links.

ALGORITHM: LiveVideoValidationAgent

PSEUDOCODE:

  FUNCTION fetch_verified_videos(topic_title, subject, grade_level):

    // Step 1: Build smart search query
    IF grade_level IN [GRADE_1, GRADE_2, GRADE_3, GRADE_4, GRADE_5]:
      query = f"{topic_title} explained for kids animated"
    ELIF grade_level IN [GRADE_6 .. GRADE_12]:
      query = f"{topic_title} {subject} class lecture explanation"
    ELIF grade_level IN [UNDERGRADUATE, POSTGRADUATE]:
      query = f"{topic_title} university lecture tutorial"
    ELSE:  // PhD
      query = f"{topic_title} research lecture academic"

    // Step 2: Call YouTube Data API
    results = youtube_api.search(
      q            = query,
      type         = "video",
      videoEmbeddable = "true",
      safeSearch   = "strict",
      maxResults   = 10
    )

    // Step 3: Filter by trusted channels
    TRUSTED_CHANNELS = [
      "Khan Academy", "3Blue1Brown", "CrashCourse", "freeCodeCamp",
      "MIT OpenCourseWare", "TED-Ed", "NPTEL", "Veritasium",
      "NumberPhile", "Physics Girl", "Professor Leonard"
    ]

    filtered = [r FOR r IN results
                IF r.channel_name IN TRUSTED_CHANNELS
                OR r.view_count > 100000]

    // Step 4: Validate each link (HTTP 200 check)
    validated = []
    FOR EACH video IN filtered[:5]:
      response = http_head(f"https://www.youtube.com/watch?v={video.id}")
      IF response.status_code == 200:
        // Step 5: Find best timestamp
        transcript = fetch_transcript(video.id)
        timestamp = find_best_timestamp(transcript, topic_title)
        video.timestamp = timestamp
        validated.append(video)

      IF len(validated) == 3:
        BREAK  // We need top 3 only

    RETURN validated[:3]

  FUNCTION find_best_timestamp(transcript, topic):
    // Semantic similarity search in transcript
    topic_embedding = embed(topic)
    best_ts = 0
    best_score = 0

    FOR EACH segment IN transcript:
      seg_embedding = embed(segment.text)
      similarity = cosine_similarity(topic_embedding, seg_embedding)
      IF similarity > best_score:
        best_score = similarity
        best_ts = segment.start_seconds

    RETURN format_timestamp(best_ts)  // "mm:ss"

          feedback_annotations.append({
            type       = "ERROR_CIRCLE",
            position   = matching_student_block.bounding_box,
            color      = "#FF0000",
            correction = expected_step.correct_text
          })
      ELSE:
        // Step not found = step skipped
        feedback_annotations.append({
          type    = "MISSING_STEP",
          message = f"Missing: {expected_step.description}",
          insert_after = previous_bounding_box
        })

    // Step 5: Compute final grade
    score_pct = (score / max_score) * 100

    IF score_pct >= 90: letter_grade = "A+"
    ELIF score_pct >= 80: letter_grade = "A"
    ELIF score_pct >= 70: letter_grade = "B"
    ELIF score_pct >= 60: letter_grade = "C"
    ELIF score_pct >= 50: letter_grade = "D"
    ELSE: letter_grade = "F"

    // Step 6: Render annotated image
    annotated_image = overlay_annotations(image_path, feedback_annotations)
    annotated_url = upload_to_storage(annotated_image)

    RETURN {
      score_pct     : score_pct,
      letter_grade  : letter_grade,
      annotated_url : annotated_url,
      feedback      : generate_feedback_text(feedback_annotations)
    }

G8. ROADMAP PRIORITY MAPPING ALGORITHM
=========================================

PURPOSE: Build the full dependency graph and determine which nodes
         unlock in what order, respecting both priority and prerequisites.

ALGORITHM: DependencyGraphBuilder

PSEUDOCODE:

  FUNCTION build_roadmap(nodes):

    // Step 1: Build adjacency list (directed graph)
    graph = {}
    in_degree = {}

    FOR EACH node IN nodes:
      graph[node.id] = []
      in_degree[node.id] = len(node.dependencies)

    FOR EACH node IN nodes:
      FOR EACH dep_id IN node.dependencies:
        graph[dep_id].append(node.id)
        // dep_id must be completed before node.id

    // Step 2: Topological Sort (Kahn's Algorithm)
    // Determines valid learning order
    queue = [node FOR node IN nodes WHERE in_degree[node.id] == 0]
    queue = sort_by_priority(queue)  // HIGH first
    sorted_order = []

    WHILE queue is not empty:
      current = queue.pop_front()
      sorted_order.append(current)

      FOR EACH neighbor_id IN graph[current.id]:
        in_degree[neighbor_id] -= 1
        IF in_degree[neighbor_id] == 0:
          neighbor = get_node(neighbor_id)
          queue.insert_sorted_by_priority(neighbor)
          // Maintains priority ordering within each "level"

    IF len(sorted_order) < len(nodes):
      RAISE CyclicDependencyError("Circular prerequisite detected!")

    // Step 3: Assign UI positions (tree layout)
    levels = assign_tree_levels(sorted_order)
    FOR EACH level, nodes_at_level IN enumerate(levels):
      FOR EACH i, node IN enumerate(nodes_at_level):
        node.ui_x = i * (NODE_WIDTH + GAP)
        node.ui_y = level * (NODE_HEIGHT + VERTICAL_GAP)

    // Step 4: Initial lock/unlock state
    FOR EACH node IN sorted_order:
      IF len(node.dependencies) == 0:
        node.status = UNLOCKED  // Root nodes are immediately available
      ELSE:
        node.status = LOCKED    // Everything else is locked initially

    RETURN sorted_order

  FUNCTION update_unlock_state(completed_node_id, all_nodes):
    // Called every time a student completes a node

    graph = build_adjacency_list(all_nodes)
    newly_unlocked = []

    FOR EACH neighbor_id IN graph[completed_node_id]:
      neighbor = get_node(neighbor_id)
      all_deps_completed = all(
        get_node(dep_id).status == COMPLETED
        FOR dep_id IN neighbor.dependencies
      )
      IF all_deps_completed:
        neighbor.status = UNLOCKED
        newly_unlocked.append(neighbor)

    save_nodes(newly_unlocked)
    RETURN newly_unlocked
